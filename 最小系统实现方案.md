# 最小系统实现方案

## 1. 文档目的

本文档给出一个面向你当前项目的 **MVP（最小可用系统）** 方案。

目标不是一次性做成“全自动、全字段、全论文”的最终平台，而是在最短路径上先实现一个能工作的系统：

- 输入：点阵/吸能 PDF 文献
- 输出：样本级结构化记录（JSON / CSV / Excel）
- 支持：人工复核
- 可持续迭代为更大的多模态信息抽取系统

该方案特别结合了你当前已有的 Excel 数据库原型，因此实现路线以“兼容现有数据结构”为第一优先级。

---

## 2. 最小系统的目标边界

## 2.1 MVP 要解决的问题

最小系统先解决以下 4 个问题：

1. 从 PDF 中提取论文基础信息
2. 从正文/表格中抽取核心结构化字段
3. 对关键图片做补充识别
4. 生成与现有 Excel 兼容的样本级记录

## 2.2 MVP 暂时不做的事

为了避免系统过重，第一版不追求：
- 全量图表精确数字化
- 全自动跨图样本对齐
- 完整知识图谱
- 模型微调
- 复杂多代理自反思闭环
- 所有子图自动准确切分

这些作为第二阶段增强项。

---

## 3. MVP 产出形式

MVP 最终应至少输出两份内容：

### A. 样本主表
格式任选其一：
- `samples_v1.jsonl`
- `samples_v1.csv`
- `samples_v1.xlsx`

### B. 证据表
- `evidence_v1.jsonl`

### C. 中间产物目录
每篇论文生成一个中间目录，例如：

```text
papers/
  2020_xxx_doi_hash/
    metadata.json
    text_chunks.json
    figures.json
    tables.json
    extraction_text.json
    extraction_figures.json
    merged_samples.json
```

这样后续便于复跑与调试。

---

## 4. MVP 的总体技术路线

最小系统建议采用 6 步主链路：

1. **PDF 解析**
2. **文本优先抽取**
3. **表格补充抽取**
4. **图像补充理解**
5. **样本级融合**
6. **规则校验 + 人工复核导出**

核心原则：

**先文本，后图像；先直接抽取，后推导；先保守入库，后逐步增强。**

---

## 5. 系统模块设计

## 5.1 模块 1：PDF 解析模块

### 功能
把 PDF 变成后续可处理的多模态对象。

### 输入
- 单篇 PDF

### 输出
- `metadata.json`
- `text_chunks.json`
- `tables.json`
- `figures.json`
- 可选：page images

### 最低要求
- 提取标题、DOI、页码
- 提取正文文本块
- 提取图 caption
- 提取表格文本
- 保存 figure 图片文件路径

### 推荐输出结构

```json
{
  "paper_id": "doi_or_hash",
  "title": "...",
  "doi": "...",
  "pages": 12,
  "text_chunks": [...],
  "tables": [...],
  "figures": [
    {
      "figure_id": "fig_1",
      "page": 4,
      "image_path": "figures/fig_1.png",
      "caption_raw": "...",
      "context": ["..."]
    }
  ]
}
```

### 为什么是第一步
因为后续所有自动抽取都不应该直接对 PDF 原文件硬解析，而应该基于结构化中间表示。

---

## 5.2 模块 2：文本优先抽取模块

### 功能
先从正文和表格里抽取最稳的字段。

### 原则
遵循优先级：
- `T1_text_table`
- `T2_derived`
- `T3_figure`

第一版 MVP 重点做 T1。

### 第一版建议抽的字段

#### 文献信息
- doi
- title
- authors_raw

#### 结构字段
- raw_design
- raw_structure
- raw_type
- is_optimized
- relative_density_raw

#### 材料字段
- raw_material
- raw_material_group
- youngs_modulus_gpa
- density_g_cm3

#### 条件字段
- analysis_type
- test_mode
- loading_direction
- velocity_m_s

#### 性能字段
- sea_j_g
- pcf_kn
- mcf_kn
- cfe
- plateau_stress_mpa

### 输出
- `extraction_text.json`

### 设计方式
推荐使用：
- 基于 schema 的 prompt 抽取
- 输出严格 JSON
- 每个字段带来源片段与置信度

### 输出示例

```json
{
  "paper_id": "...",
  "samples": [
    {
      "sample_label_in_paper": "Sample A",
      "raw_design": "Gyroid",
      "raw_type": "TPMS",
      "raw_material": "316L",
      "relative_density_raw": "0.25",
      "sea_j_g": 18.6,
      "source_type": "text",
      "evidence": {
        "chunk_id": "chunk_12",
        "text": "..."
      },
      "confidence": 0.86
    }
  ]
}
```

---

## 5.3 模块 3：表格补充抽取模块

### 功能
从论文表格中补充文本模块没抽全的字段。

### MVP 中的定位
它仍属于 T1 层，只是输入源不同。

### 适合优先从表格抽取的字段
- 相对密度
- 材料
- 工艺
- 样品编号
- SEA
- PCF
- MCF
- Plateau stress
- 测试速度/应变率

### 注意事项
- 表头多层级时不要急着做复杂结构还原
- 第一版只要把“单元格内容 + 上下文表头”保留下来即可

### 输出
- `extraction_tables.json`

---

## 5.4 模块 4：图像补充理解模块

### 功能
在文本/表格没有给够信息时，对图像做“保守补充”。

### MVP 中不建议一开始做的事情
- 不要一上来就追求图中曲线精确点读
- 不要一上来做所有复杂子图分离

### MVP 中建议做的事情
先做“图像语义补充”而不是“图像精确数字化”：
- 判断图类型
- 判断是否与结构相关
- 判断是否与性能曲线相关
- 生成图像摘要
- 提取图中显式可读文字（坐标轴、图例、样本名）

### 图类型建议先分 5 类
- `structure_diagram`
- `curve_plot`
- `experiment_setup`
- `microstructure_image`
- `composite_figure`

### 输出
- `extraction_figures.json`

### 输出示例

```json
{
  "figure_id": "fig_3",
  "figure_type": "curve_plot",
  "related_fields": ["sea_j_g", "plateau_stress_mpa"],
  "summary": "Figure shows stress-strain curves of gyroid and diamond lattices under quasi-static compression.",
  "candidate_sample_labels": ["Gyroid", "Diamond"],
  "confidence": 0.71
}
```

---

## 5.5 模块 5：样本级融合模块

### 功能
将文本、表格、图像三个来源的结果融合成样本级记录。

### 为什么这是核心模块
因为真正要入库的是“样本”，不是“文本片段”或“一张图”。

### 融合规则（MVP 版）
1. 先以文本抽取为主骨架
2. 用表格补值
3. 用图像只做补充或确认
4. 同字段冲突时采用优先级：
   - text/table > derived > figure
5. 保留冲突记录到 review 字段

### 输出
- `merged_samples.json`

### 样本主键建议

```text
sample_id = doi + sample_label + normalized_structure + row_index
```

如果没有 sample_label，就先生成临时 ID。

---

## 5.6 模块 6：规则校验与导出模块

### 功能
在入库前做基础质量控制。

### 最低限度要做的校验

#### 字段完整性检查
- doi 是否存在
- title 是否存在
- 至少一个结构字段是否存在
- 至少一个性能字段是否存在

#### 值域检查
- `is_optimized` 必须是 true/false/null
- `analysis_type` 必须在枚举内
- `test_mode` 必须在枚举内
- `loading_direction` 必须在枚举内

#### 数值合理性检查
- 相对密度通常应在 0~1 之间
- SEA、PCF、MCF、plateau stress 应为非负
- velocity_m_s 应为非负

#### 标准化检查
- Material 是否已映射为 canonical
- Process 是否已映射为 canonical
- Type 是否已映射为结构主类/子类

### 输出
- `samples_v1_clean.jsonl`
- `samples_v1_review.csv`

其中 review 表只放需要人工看的一部分。

## 6. 最小系统与现有 Excel 的对接方式

当前 Excel 应作为 MVP 的四个输入/输出接口：

### 6.1 作为字段模板
MVP 抽取字段应优先兼容你现有 27 个有效字段。

### 6.2 作为标准化映射来源
- 结构映射表从 `Design / Structure / Type` 出发
- 材料映射表从 `Material / Material.G` 出发
- 工艺映射表从 `Process` 出发

### 6.3 作为人工复核界面
自动抽取结果导出到和现有表格相似的结构，便于你人工修正。

### 6.4 作为初始评估集
你现有人工整理的样本记录可作为第一批 gold 标注。

## 8. 开发阶段划分

## 阶段 A：先打通单篇论文流程

### 目标
给 1 篇 PDF，系统能输出结构化 JSON。

### 交付
- PDF 解析
- 文本抽取
- 基础融合
- JSON 导出

### 完成标准
- 不要求字段很全
- 但输出必须结构稳定
- 必须可以人工检查

---

## 阶段 B：扩展到 10~20 篇论文

### 目标
验证系统是否能跨论文工作。

### 交付
- 批处理脚本
- review 表
- 基础映射表

### 完成标准
- 能稳定生成 `samples_v1.csv`
- 能定位需要人工复核的记录

---

## 阶段 C：加入图像增强

### 目标
对文本缺失较多的论文提升抽取覆盖率。

### 交付
- 图类型识别
- 图像摘要模块
- 图像候选字段补充

---

## 阶段 D：再做高级增强

后续再考虑：
- 曲线点数字化
- 子图切分
- 多代理协同
- 微调
- 检索系统
- 功能需求到候选结构反查

## 10. MVP 的推荐执行顺序

### 第 1 周
- 定 schema v1
- 清洗现有 Excel
- 建立 mapping 表

### 第 2 周
- 实现 PDF 解析
- 实现文本抽取
- 输出单篇 JSON

### 第 3 周
- 实现表格补充抽取
- 实现融合与导出
- 跑 10~20 篇论文

### 第 4 周
- 实现图像摘要模块
- 增强 review 机制
- 输出 MVP 版本数据库

---

## 11. 成功标准

如果 MVP 达到下面 5 点，就说明路线是正确的：

1. 单篇论文能稳定输出 JSON
2. 多篇论文能批量输出样本主表
3. 输出字段与现有 Excel 兼容
4. 冲突和缺失会被明确标记，而不是悄悄出错
5. 人工复核后能顺利回写数据库

---

## 12. 本阶段最重要的结论

你当前最应该做的，不是先追求一个“超强识图模型”，而是先把最小系统做成：

**PDF 可解析、字段可抽取、样本可融合、结果可复核。**

真正能推动项目向前的 MVP 不是“全自动”，而是：

**半自动但结构正确、可追溯、可持续扩展。**
