# 项目长期路线与验收治理设计

## 1. 目标

本设计用于为 `C:\Users\Administrator\Desktop\dianzhen shujuku` 建立一套长期任务治理框架，用来同时回答三个问题：

1. 这个项目长期要做成什么。
2. 当前阶段在做什么，以及 `2026-04-21` 产品验收要验什么。
3. 后续哪些工作包可以新开支线，哪些必须在同一条执行支线推进。

本轮目标不是继续扩展业务功能，也不是开发新的自动化工具，而是先把项目总目标、阶段路线、验收对象和分支策略固定下来，避免后续开发跑偏。

## 2. 范围与非范围

### 2.1 本轮范围

本轮只定义并落地以下治理资产：

- 长期任务主目标与阶段路线
- `2026-04-21` 的阶段里程碑验收定义
- 可视化的分支拆分策略
- 仓库内总览入口文件
- 仓库外长期任务记忆骨架

### 2.2 本轮非范围

本轮明确不包含以下内容：

- 业务功能实现
- 抽取脚本开发
- review 工具开发
- UI 或可视化系统开发
- 自动化任务调度脚本
- 全量文献导入与数据跑通

## 3. 第一性原则

### 3.1 长期任务本体必须是项目本体

长期任务不能定义成 `dispatch/tasks` 这套调度体系本身，也不能定义成单次验收。  
长期任务本体必须是整个 `dianzhen shujuku` 项目的产品路线与研发路线。

### 3.2 `2026-04-21` 验收是里程碑，不是终局

明天的验收不是验最终数据库产品是否完成，而是验项目是否已经进入“可控开发状态”。  
因此，明天的验收应被定义为当前阶段里程碑，而不是长期任务本体。

### 3.3 调度体系服务项目，而不替代项目

`dispatch/tasks` 是执行调度层，只负责：

- 任务分配
- 状态维护
- 分支管理
- 流程约束

它不负责回答“项目最终要做什么”，也不单独承担长期路线定义。

### 3.4 一眼看清比信息齐全更重要

长期治理文件首先必须让人一眼看清：

- 当前项目在做什么
- 当前阶段是什么
- 明天验什么
- 下一步从哪开干
- 哪些任务可拆支线

只有在这个前提下，详细信息才有价值。

## 4. 治理结构

本项目的长期治理固定为三层：

### 4.1 项目主目标层

回答：

- 项目最终目标是什么
- 当前明确不做什么
- 当前阶段为什么存在

### 4.2 阶段路线层

回答：

- 项目分哪几个阶段推进
- 当前在哪个阶段
- 当前阶段的完成标志是什么
- 最近里程碑是什么

### 4.3 执行编排层

回答：

- 当前有哪些工作包
- 哪些工作包必须留在 `dispatch/tasks`
- 哪些工作包必须同一条支线推进
- 哪些工作包满足前置条件后可新开支线

## 5. `2026-04-21` 验收定义

### 5.1 里程碑命名

将 `2026-04-21` 的产品验收定义为：

**M0 可控开发启动验收**

### 5.2 验收对象

本次里程碑验收要检查的是：

1. 项目目标是否清楚
2. 当前骨架是否可操作
3. 数据契约是否成形
4. 下一步开发入口是否一眼看清

### 5.3 本次验收不检查

本次验收明确不检查以下内容：

- 最终数据库产品是否完成
- 全量文献是否已导入
- 抽取脚本是否全部开发完成
- 最终 UI 或完整系统是否可演示

## 6. 阶段路线

项目路线固定为四段：

### 6.1 P0 / M0：可控开发启动阶段

目标：

- 项目目标与边界清楚
- 仓库骨架与调度体系可用
- schema / mapping 边界稳定
- 下一步工作包与分支策略明确

里程碑：

- `2026-04-21`：M0 可控开发启动验收

### 6.2 P1：最小贯通链路阶段

目标：

- 先跑通第一条从输入到输出的最小数据链路
- 固化首条真实开发接口与 review 边界

### 6.3 P2：并行能力扩展阶段

目标：

- 输入适配、抽取增强、review / export 能力按模块拆分并行推进

### 6.4 P3：集成交付阶段

目标：

- 将并行成果重新收束为稳定交付物

## 7. 工作包与分支策略

### 7.1 必须留在 `dispatch/tasks`

以下资产必须留在 `dispatch/tasks` 上维护，不开执行支线：

- 项目总览
- 阶段路线
- 分支策略
- 任务看板
- 分支映射
- 调度规则
- workflow
- 任务模板
- 当前项目状态总览

### 7.2 必须同一条执行支线推进

以下工作包必须在同一条执行支线内连续推进：

#### 数据契约包

- `schemas/`
- `mappings/`
- 校验口径
- review 判定边界

原因是它们共同定义“什么叫一条有效数据”，不适合拆成并行支线分别推进。

#### 首条最小贯通链路包

- 输入接入
- 解析中间层
- 抽取结果落盘
- sample / evidence 对齐
- 最小 review / export 闭环

原因是第一条链路在接口未跑通前，不适合拆分成并行子支线。

### 7.3 满足条件后可新开支线

只有在输入输出契约稳定、写入文件集合不重叠、验收口径清楚后，以下工作包才允许新开执行支线：

#### 输入适配类

- PDF 输入适配
- Excel 种子表适配
- 原始文件组织规则增强

#### 抽取能力类

- 文本抽取
- 表格抽取
- 图像 / 图注相关抽取
- 字段级抽取增强

#### review / 清洗辅助类

- review 队列增强
- 人工复核辅助文档
- 质量检查脚本
- 清洗辅助工具

#### 导出 / 交付类

- JSON 导出
- CSV / XLSX 导出
- 验收展示材料
- 汇总报告视图

## 10. 展示原则

为保证“看一眼就知道项目在做什么”，所有总览资产必须遵守以下规则：

1. 第一屏先写当前阶段、当前目标、最近里程碑、当前下一步
2. 分支信息只按“留在 dispatch/tasks / 同支线推进 / 可新开支线”表达
3. 阶段路线控制在四段以内
4. 一个文件只回答一个问题，不混用

## 13. 结论

本设计的核心不是新增更多流程，而是把以下三件事固定下来：

1. 项目长期目标是什么
2. 当前阶段在做什么
3. 后续开发怎样拆支线才不会跑偏

只要这三件事被固定，明天的验收就会从“看材料”变成“看项目是否已经进入可控开发状态”。
