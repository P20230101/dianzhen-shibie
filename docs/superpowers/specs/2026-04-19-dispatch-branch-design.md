# Git Dispatch 分支模型设计

## 1. 目标

本设计用于为 `C:\Users\Administrator\Desktop\dianzhen shujuku` 定义一套长期可用的 Git 分支工作流。

本轮目标不是开始业务开发，而是先建立一个逻辑正确的仓库与分支模型，满足以下约束：

1. 当前目录纳入 Git 仓库管理。
2. 存在一条长期任务分配支线。
3. 后续所有开发支线都从该任务分配支线切出。
4. 任务分配支线只维护任务调度资产，不承载业务代码。
5. 主分支只接收真正需要进入主线的开发成果。

## 2. 本轮范围与非范围

### 2.1 本轮范围

本轮只定义：

- 仓库初始化后的主分支角色
- 长期任务分配支线的角色
- 子开发支线的切分规则
- 子开发支线回到主线的集成规则
- 任务调度资产在仓库中的目录结构
- 禁止行为与最小验证规则

### 2.2 本轮非范围

本轮不定义：

- 任何业务功能开发
- 具体任务内容本身
- CI/CD 流程
- GitHub PR 模板
- 多人权限策略
- 自动化发布策略

## 3. 设计原则

### 3.1 任务调度与业务开发分层

任务分配支线只负责“派工与状态管理”，业务代码开发必须发生在子开发支线上。

### 3.2 主线只接收代码成果

`main` 不能被任务调度提交污染。任何把任务分配资产带入主线的默认集成方式都视为错误流程。

### 3.3 从父支线切出，不等于回父支线集成

子开发支线可以从 `dispatch/tasks` 切出，但不意味着最终要普通 merge 回 `dispatch/tasks` 或 `main`。

### 3.4 显式集成，不做默认 merge

从 `dispatch/tasks` 派生出来的开发支线，进入 `main` 时必须先剥离调度父线影响，不能依赖普通 merge。

### 3.5 派工资产单独治理

任务看板、任务分配记录、分支映射和归档必须集中保存在调度支线上指定目录中。

## 4. 分支拓扑

### 4.1 `main`

作用：

- 稳定代码主线
- 只接收要进入主线的真实开发成果
- 不维护任务分配状态

### 4.2 `dispatch/tasks`

作用：

- 长期存在的任务分配支线
- 只维护任务拆解、分配、状态、分支指派与调度规则
- 作为所有实际开发支线的父分支

硬约束：

- 不在该分支上做业务代码开发
- 不在该分支上堆积功能实现提交

### 4.3 子开发支线

作用：

- 承载实际开发工作
- 全部从 `dispatch/tasks` 切出

命名建议：

- `feat/<topic>`
- `fix/<topic>`
- `data/<topic>`
- `exp/<topic>`

## 5. 集成规则

### 5.1 `dispatch/tasks` 的提交范围

`dispatch/tasks` 只允许这类提交：

- 任务拆解
- 任务状态更新
- 分支分配记录
- 调度说明文档
- 任务归档

不允许：

- 业务代码开发
- schema 或实现文件的功能性修改
- 为了省事把开发成果直接堆回调度支线

### 5.2 禁止的集成方式

以下方式均视为错误流程：

1. 直接把 `dispatch/tasks` merge 到 `main`
2. 从 `dispatch/tasks` 切出的开发支线直接普通 merge 到 `main`
3. 在 `dispatch/tasks` 上直接写业务代码再合回 `main`

原因：

开发支线的共同祖先是 `dispatch/tasks`。如果直接普通 merge，则 `dispatch/tasks` 上的任务调度提交也会一并进入 `main`。

### 5.3 允许的集成方式

只允许两种方式把开发成果带入 `main`：

#### 方式 A：`rebase --onto`

用途：

- 把某个开发支线相对于 `dispatch/tasks` 新增的开发提交整体搬到 `main` 之上

适用场景：

- 开发提交连续、干净、适合整体迁移

#### 方式 B：`cherry-pick`

用途：

- 从 `main` 新开干净集成支线，只摘取实际开发提交

适用场景：

- 开发支线历史较乱
- 只需要带入部分提交

### 5.4 推荐顺序

推荐优先级：

1. 优先使用 `rebase --onto`
2. 无法整洁迁移时，使用 `cherry-pick`

## 6. 仓库内任务调度资产结构

任务调度支线建议维护如下目录：

```text
tasks/
  board.md
  branches.md
  rules.md
  assignments/
  archive/
```

### 6.1 `tasks/board.md`

作用：

- 总任务看板
- 维护任务状态，如 `todo / doing / blocked / done`

### 6.2 `tasks/branches.md`

作用：

- 建立任务编号与开发支线的对应关系
- 记录每个任务从哪个分支切出、准备如何回主线

### 6.3 `tasks/rules.md`

作用：

- 固定说明如何切分支
- 固定说明如何回主线
- 固定说明哪些内容允许进入 `dispatch/tasks`

### 6.4 `tasks/assignments/`

作用：

- 每个任务独立保存分配记录
- 记录负责人、目标、依赖、状态、目标开发支线

### 6.5 `tasks/archive/`

作用：

- 归档已完成或废弃任务
- 避免看板长期膨胀

## 7. 最小操作规范

初始化完成后，后续工作流固定如下：

1. 在 `dispatch/tasks` 上维护或更新任务调度信息
2. 从 `dispatch/tasks` 切出具体开发支线
3. 在开发支线上完成业务实现
4. 开发完成后，不直接普通 merge 到 `main`
5. 通过 `rebase --onto` 或 `cherry-pick` 将真实开发提交带入 `main`

## 8. 错误处理规则

本轮只接受显式正确流程，不接受“先做了再说”的补丁式操作：

- 如果开发支线已经混入任务调度提交，必须先清理历史，再考虑回主线
- 如果有人在 `dispatch/tasks` 上直接写了业务代码，该分支不再视为纯调度支线，必须人工处理
- 如果无法明确哪些提交属于任务调度、哪些属于开发实现，禁止继续集成

## 9. 最小验证规则

本轮落地后，至少要验证：

1. 当前目录已经是 Git 仓库
2. 存在 `main`
3. 存在 `dispatch/tasks`
4. 存在 `tasks/` 调度目录骨架
5. 可以从 `dispatch/tasks` 成功切出一个测试开发支线
6. `main` 与 `dispatch/tasks` 的提交职责清晰可区分

## 10. 本轮结论

这套模型的核心不是“多一条分支”，而是明确分出两种完全不同的职责：

1. `dispatch/tasks` 负责派工
2. 子开发支线负责做事
3. `main` 负责接收真正需要进入主线的代码成果

后续实现必须严格遵守以下底线：

- 所有开发支线都从 `dispatch/tasks` 切出
- `dispatch/tasks` 不做业务代码开发
- 开发成果进入 `main` 前必须剥离调度父线影响
