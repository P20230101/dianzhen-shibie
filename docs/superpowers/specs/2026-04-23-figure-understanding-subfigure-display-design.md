# 图像理解层子图展示设计

## 背景

当前图像理解层已经能把真实 PDF 的 figure 裁图和结构化解释放到静态 HTML 报告里，用户也已经确认用 Chrome 直接查看真实裁图作为主要验收方式。现有报告页以“总图卡片”为单位展示 `figure_type`、`recaption`、`figure_summary`、`panel_labels` 和 `subfigure_map`，但子图结果仍然混在总图卡片内部，没有形成清晰的视觉分区。

## 目标

把单个总图中的子图结果单独展开显示，让用户一眼看清：

- 这张总图是什么
- 这张图包含哪些子图
- 每个子图对应什么解释

本次只做**网页展示层**的拆分，不把子图拆成新的数据记录，不改 `figures_v1.jsonl` 的总体结构，也不改图像理解层的抽取逻辑。

## 方案

采用已确认的方案 A：

- 每张 figure 仍然保留一张总图卡片
- 在总图预览下方新增一个独立的 `Subfigures` 区块
- `Subfigures` 区块按子图条目逐行展示
- 每个子图条目展示：
  - 子图标签
  - 子图解释文本
  - 必要时保留与总图的对应关系说明

如果某张图没有明确的 `subfigure_map`，则只显示 `panel_labels`，并明确标成“无可展开子图说明”。

## 页面结构

单张 figure card 内部按以下顺序排列：

1. 总图预览
2. 总图级结构化结果
3. `Subfigures` 区块
4. 状态标识和边界说明

`Subfigures` 区块要视觉上和总图结果分开，避免用户把子图解释误认为总图摘要。

## 数据规则

网页只读现有 figure 记录中的字段：

- `panel_labels`
- `subfigure_map`
- `figure_type`
- `recaption`
- `figure_summary`
- `needs_manual_review`

不引入新字段，不要求子图在数据层变成独立记录。

渲染规则：

- `subfigure_map` 有内容时，优先展示映射结果
- `subfigure_map` 为空时，回退到 `panel_labels`
- 两者都为空时，显示空状态，不隐藏该区块

## 验收标准

这次改动完成后，用户应能在 Chrome 中直接看到：

- 每张总图仍然完整保留
- 子图解释单独成区块显示
- 子图结果不再和总图摘要混写在一起
- 真实 PDF 裁图和子图解释之间的对应关系更清楚

## 非目标

本次不做：

- 子图新建数据记录
- 搜索和筛选
- 多页面报告
- 在线服务化
- 重新抽取图像内容
