# 可直接安装使用的工具清单 + 项目推荐技术栈

## 1. 文档目的

本文档用于回答一个非常实际的问题：

**在“点阵 / 吸能文献数据库构建”项目中，PDF 解析、图片解析、曲线读取、图文对齐、多模态抽取这些环节，到底有哪些工具可以直接利用？哪些只是论文里的思路，暂时不适合直接上手？**

本文将把相关论文中出现的工具、框架与模型，整理成一份面向实施的工具文档，重点区分：

1. **可直接安装和使用的工具**
2. **可以直接调用的现成模型 / API**
3. **论文中有效但更适合作为后期模块的技术**
4. **不建议一开始就投入的方向**

---

## 2. 结论先看：最值得直接用的工具

如果目标是尽快做出一个可运行的最小系统（MVP），推荐优先使用下面这组：

### 第一优先级（建议立刻开始）

1. **Docling**
   - 用途：PDF 解析、正文抽取、表格抽取、页面图像导出
   - 角色：整条系统的 **PDF 解析入口**
   - 建议：作为第一步基础设施

2. **Qwen2.5-VL / GPT-4o / Gemini（任选其一或组合）**
   - 用途：科学图像理解、图像重描述（recaption）、字段候选抽取
   - 角色：整条系统的 **图像理解引擎**
   - 建议：先直接用，不要一开始自己训练

3. **PlotDigitizer**
   - 用途：手工 / 半自动提取曲线图数据点
   - 角色：高质量曲线 gold data 构建工具
   - 建议：先用于关键图精确抽点和标注对照

### 第二优先级（MVP 跑通后加入）

4. **Uni-Parser + PaddleOCR**
   - 用途：高质量 figure-caption-context 对齐、subfigure 切分、panel 标识识别
   - 角色：图文绑定增强模块

5. **YOLO**
   - 用途：图、表、示意图区域检测；后续可做版面区域检测与图内元素检测
   - 角色：视觉检测模块

6. **DePlot**
   - 用途：曲线图 / 图表转线性化表格
   - 角色：专用 chart-to-table 增强工具
   - 建议：后期用于图表专项增强，不要作为全系统主线

---

## 3. 论文中直接提到、且对你项目有现实参考价值的工具

### 3.1 Docling

#### 文献依据
SciEx 将 **Docling** 作为 PDF Extractor 使用，用来解析 PDF、抽取文本和视觉内容，并将 text、tables、figures 组织为后续可检索和抽取的多模态数据库。论文明确写到：Docling 是一个开源 document conversion toolkit，支持细粒度布局分析与结构识别，还支持导出 HTML、XML、Markdown 等格式。fileciteturn18file0

#### 你可以直接拿来做什么
- 解析 PDF 正文
- 按段落 / 章节切分文本
- 抽取表格
- 导出页面图片
- 给后续图像模型提供图像输入源
- 为样本级数据库准备 paper-level 原始结构数据

#### 优点
- 开源
- 更偏“工程可用”而不是纯论文 demo
- 适合你现在的 **PDF → 结构化中间层** 这一步
- 和你后续 schema 抽取路线兼容性高

#### 局限
- 它解决的是“拆文档”，不是“自动懂科学图”
- figure-caption-context 的绑定精度不一定足够覆盖所有复杂复合图场景

#### 适合你的阶段
**现在就适合上手。**

---

### 3.2 Uni-Parser + PaddleOCR

#### 文献依据
OmniScience 在 figure-caption-context triplet 构建中，使用 **Uni-Parser** 做科学文档解析，具体包括：
- layout detection
- figure-caption group 定位
- subfigure localization
- legend extraction

同时使用 **PaddleOCR** 识别子图中的 a / b / c 之类 panel 标识，再将 sub-caption 和 panel 进行匹配。论文还指出，Uni-Parser 对异构 PDF 版式下的 figure-caption-context 绑定尤其关键。fileciteturn17file13 fileciteturn18file2

#### 你可以直接拿来做什么
- 提升 figure 与 caption 的绑定质量
- 解析复合图中的子图 panel
- 构建 `figure + caption + context + subfigure` 的高质量中间数据
- 识别图中 a/b/c/d 的 panel 标签

#### 优点
- 很适合你未来想做的 **图文证据链**
- 对复合图、跨栏、复杂排版更有价值
- 比简单 PDF 转文本更贴近你后续的图像识别需求

#### 局限
- 工程复杂度高于 Docling
- 不建议作为第一天就必须跑起来的唯一入口
- 更适合作为 **第二阶段增强版面解析工具**

#### 适合你的阶段
**MVP 跑通后尽快加入。**

---

### 3.3 PlotDigitizer

#### 文献依据
MATVIX 在构建 Polymer Biodegradation 数据时，明确提到专家使用 **PlotDigitizer** 对论文中的曲线进行 tracing，通过坐标轴标定将图中的数据点转换为结构化 JSON。fileciteturn15file2

MATVIX 还专门指出：像 PlotDigitizer 这样的工具虽然可以提取曲线点，但缺乏自动化，需要人工校准坐标轴和标注数值；他们认为目前并没有真正成熟的全自动方案。fileciteturn17file5

#### 你可以直接拿来做什么
- 提取应力–应变曲线
- 提取能量吸收曲线
- 提取 SEA / peak stress / plateau stress 对应曲线
- 制作 gold standard 曲线数据
- 给后续自动模型建立人工对照集

#### 优点
- 立刻可用
- 适合做高质量少量标注
- 非常适合作为“最早可落地的数据获取工具”

#### 局限
- 偏人工 / 半自动
- 不适合大规模全自动批处理
- 对图像质量、坐标轴清晰度比较敏感

#### 适合你的阶段
**现在就能用，而且很有价值。**

---

### 3.4 Mathpix

#### 文献依据
MATVIX 在实验流程中，先把 PDF 转成 LaTeX / TeX，再基于这个结构化文档做后续抽取。论文明确写到他们使用 **Mathpix** 完成 PDF → LaTeX 转换。fileciteturn17file0

但 MATVIX 同时也说明了一个重要发现：
- 把整篇 TeX 文件和所有图片一起喂给 VLM，效果并不好
- 成本也较高
- 因此他们改成：文本先抽取、逐图扩展、最后融合 的多步流程fileciteturn17file0

#### 你可以直接拿来做什么
- 对少量重点文献做高质量结构化文本转换
- 获取更规整的 section / subsection / image 结构
- 作为 Docling 的补充对照工具

#### 优点
- 对数学公式、版面文本较友好
- 转成 TeX 后便于后续 NLP 处理

#### 局限
- 不建议作为你全系统的唯一依赖
- 费用和批量处理上要考虑成本
- 仍然不能替代 figure-level 专项解析

#### 适合你的阶段
**可选使用，适合重点论文精处理，不建议做第一主线。**

---

### 3.5 DePlot

#### 文献依据
MATVIX 把 **DePlot** 视为一个 specialized model，用于图表类图像的信息提取。他们的实验表明：在最优配置中引入 DePlot，能提升曲线抽取相关指标。fileciteturn17file5

但 MATVIX 也明确指出：DePlot 更擅长的是 **simple plot-to-table conversion**，而对于“长文档 + 文本 + 表格 + 图形强耦合”的复杂多模态抽取，它并不够。fileciteturn18file1

#### 你可以直接拿来做什么
- 单张曲线图转线性表格
- chart / plot 的局部增强处理
- 后期作为曲线图专项模块

#### 优点
- 对图表解析有针对性
- 适合作为曲线图增强器

#### 局限
- 不是全流程解决方案
- 不能替代全文抽取与多模态融合
- 不适合一开始当核心主干

#### 适合你的阶段
**后期加入。**

---

### 3.6 YOLO

#### 文献依据
nanoMINER 明确写到，他们使用 **YOLO** 进行 visual data extraction，用于检测图像中的 figures、tables、schemes 等对象，再由 GPT-4o 结合文本描述进行进一步解释和融合。fileciteturn17file17

#### 你可以直接拿来做什么
- 页面中的图 / 表区域检测
- 结构图区域定位
- 图内 legend / ROI / 局部区块检测
- 后续建立图像区域训练集

#### 优点
- 工程生态成熟
- 很适合做区域检测前处理
- 方便后面和你自己的标注数据对接

#### 局限
- YOLO 本身只解决“在哪里”，不解决“这是什么语义”
- 需要训练或适配，才能更贴合你的论文页面场景

#### 适合你的阶段
**第二阶段 / 第三阶段更合适。**

---

### 3.7 Qwen2.5-VL / GPT-4o / Gemini

#### 文献依据
OmniScience 直接使用多模态大模型做动态模型路由和 recaption，包括 Gemini、GPT 系列与 Qwen3-VL 系列；同时用 Qwen2.5-VL-3B 做微调验证，证明科学图像理解能力可被增强。fileciteturn18file2

MATVIX 也系统评测了 GPT-4o、Gemini、Qwen2.5-VL-7B / 72B 等模型在材料科学文献抽取中的表现。fileciteturn17file0

#### 你可以直接拿来做什么
- 结构图类型识别
- 图像重描述（recaption）
- 图中候选字段抽取
- 图文联合补全
- 图像 QA / caption QA

#### 优点
- 不需要你先训练模型
- 非常适合做 MVP 中的图像理解层
- 与 Docling / Uni-Parser 输出容易结合

#### 局限
- 成本 / API / 本地部署能力需要平衡
- 输出需要 schema 约束与规则校验，否则不够稳

#### 适合你的阶段
**现在就能直接利用。**

## 4. 哪些是“思路很有价值，但不建议一上来就重投入”的

### 4.1 LayoutLM / LayoutLMv2 / MatSciBERT 这一类
MATVIX 明确指出，这些模型虽然在 visually rich documents 上做过很多工作，但它们更适合较短上下文、较简单的实体识别 / 关系抽取，而不适合长文档、复杂层级 JSON 重建、多图联合推理。fileciteturn17file6

**结论：**
不建议你现在把精力投入在这类传统模型微调上。

### 4.2 “整篇 PDF + 所有图片一次性喂给 VLM”
MATVIX 的实验证明，这样做效果并不好，成本也高。更好的方式是：
1. 文本先抽
2. 单图补充
3. 最后融合fileciteturn17file0

**结论：**
不要走“大一统单步抽取”的路线。

## 5. 推荐技术栈（按你的项目来）

## 5.1 MVP 最小系统技术栈

### 目标
先做一个能跑通 20–50 篇论文、输出样本级 JSON/Excel 的系统。

### 推荐组合
- **PDF 解析**：Docling
- **图像理解**：Qwen2.5-VL 或 GPT-4o
- **曲线人工高精抽取**：PlotDigitizer
- **结果存储**：JSON + Excel
- **规则校验**：Python + schema 校验规则

### 这一阶段不做
- 不先训练 YOLO
- 不先训练专用视觉模型
- 不先做大规模自动曲线抽点

---

## 5.2 第二阶段增强技术栈

### 目标
提高图文绑定质量、复合图处理能力、图像字段准确率。

### 推荐加入
- **Uni-Parser**：高质量图文绑定
- **PaddleOCR**：panel 标识识别
- **YOLO**：图表区域检测
- **VLM 专项 prompt**：按图类型分别处理

---

## 5.3 第三阶段研究型技术栈

### 目标
做你自己的点阵文献多模态抽取系统。

### 推荐方向
- 用 OmniScience 思路构建 `figure-caption-context-subfigure` 数据
- 用 Qwen2.5-VL 做领域适配 / 微调
- 加入结构图主类 / subtype 分类器
- 加入曲线图自动读取模块
- 加入 evidence log 与 T1/T2/T3 来源标注

---

## 6. 你的项目里，每个工具最适合放在哪一层

### 层 1：PDF 解析层

- Uni-Parser
- PaddleOCR
- YOLO

- Qwen2.5-VL

- Gemini

- PlotDigitizer

## 7. 最终建议：怎么选，不会走弯路

### 如果你现在只想“先做出来”
用这套：

- Docling
- Qwen2.5-VL 或 GPT-4o
- PlotDigitizer
- JSON / Excel

### 如果你想“图文绑定更强”
在上面的基础上加：

- Uni-Parser
- PaddleOCR

### 如果你想“以后做成论文系统”
再继续加：

- YOLO
- DePlot
- 领域微调 VLM

## 9. 直接可执行的起步方案

### 第一步
用 Docling 批量解析 10 篇点阵吸能论文，输出：
- 章节文本
- 表格
- 图像
- 页码映射

### 第二步
对每篇论文的 figure 建一个中间表：
- figure_id
- page
- caption
- context
- image_path
- figure_type

### 第三步
用 Qwen2.5-VL 或 GPT-4o 对每张图生成：
- 图像描述
- 是否与结构/性能/实验条件相关
- 候选字段

### 第四步
对最关键的应力–应变图和能量吸收图，用 PlotDigitizer 手工抽点，形成你的 gold 数据。

### 第五步
把以上结果写入样本级 JSON / Excel，开始做数据库原型。

---

## 10. 最后一句话建议

对于你现在这个项目，最稳的工具路线不是“先训模型”，而是：

**Docling 先拆论文，VLM 先看图补语义，PlotDigitizer 先保关键曲线质量，Uni-Parser 后续再补高精图文绑定。**

也就是说：

**先搭工程链路，再追求模型上限。**
